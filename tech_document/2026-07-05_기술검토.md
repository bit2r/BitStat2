- **검토 범위**: 프로젝트 구조, 빌드/배포, 콘텐츠(Shiny/shinylive) 코드, 설정, 문서화, 라이선스

## 1. 프로젝트 개요

- **성격**: 통계 비전공자·일반인을 위한 오픈 통계 학습 사이트. Quarto `website` 프로젝트로 빌드하여 GitHub Pages(`docs/`)로 배포.
- **구성**: 7개 대주제(01 데이터 ~ 07 이론) 아래 23개 학습 모듈. 각 모듈은 `index.qmd` + `shiny/app.R`(+ 썸네일)로 구성.
- **인터랙티브 방식**: `shinylive` 필터로 브라우저(WASM)에서 Shiny 앱을 직접 실행. `webr` 필터도 활성화.
- **규모**: 저장소 약 119MB(그중 `docs/` 77MB), 콘텐츠 `index.qmd` 23개, `app.R` 23개.

## 3. 문제점 / 리스크 (Issues)

심각도: 🔴 높음 / 🟠 중간 / 🟡 낮음

### 3.1 코드 중복 및 실측 drift — 🔴 높음 ✅확정
- **현상**: 23개 모듈 중 22개가 `shiny/app.R`와 `index.qmd` 내 `shinylive-r` 블록에 **동일 계열 앱 코드를 이중 보관**. (`corr_reg`는 shinylive 블록 2개)
- **근거(전수 diff, 공백 정규화)**: 23개 모듈 인라인 블록 vs `shiny/app.R` 비교 결과 — **IDENTICAL 15개 / DRIFT 8개 / app.R 누락 0개**. DRIFT 모듈:
  - `02_eda/univariate_stat`: **차이 151줄 — 실질적 대규모 divergence**.
  - `07_theory/lln`: 차이 18줄.
  - `01_data/sampling_method`: 인라인에 `library(showtext); showtext_auto()` 있으나 `app.R`엔 **없음**(폰트 누락).
  - `07_theory/mle`, `05_infer/clt`, `02_eda/desc_stat`, `02_eda/wordcloud`, `04_testing/x_score`: 각 1~2줄(주로 폰트/공백).
- **핵심**: 실제 배포되는 것은 `index.qmd`의 **인라인 shinylive 블록**이며, `shiny/app.R`는 8개 모듈에서 이미 어긋난 **stale 사본**(특히 `univariate_stat`는 사실상 다른 앱). 나머지 15개는 아직 일치하나 정본 규칙이 없어 언제든 어긋날 수 있음.
- **영향**: `shiny/app.R`를 정본으로 착각해 수정하면 배포본에 반영되지 않음. 유지보수 비용 2배 + 오작업 위험. 어느 쪽이 정본인지 문서화 부재.

### 3.2 한글 폰트 처리 비일관 — 🔴 높음 ✅부분확정 (렌더링 정확성)
- **현상(배포되는 인라인 블록 기준)**: 23개 `index.qmd` 중 **15개만** `showtext` 사용, **8개는 폰트 설정 없음**(`dist_continuous`, `dist_discrete`, `dist_fitting`, `multivariate_stat`, `wordcloud`, `esquisse`, `viz`, `lln`).
- **폰트 지정 제각각**:
  - `x_score`: 인라인 블록에 `family = "Tahoma"`가 **4곳** + showtext 혼용 — **Tahoma는 Windows 전용, 한글 미지원이며 WASM 런타임에 부재**.
  - `one_mean`/`one_prop`/`two_means`/`two_prop`: `base_family = "Nanum Gothic"`.
  - `font_add_google("Nanum Gothic")`이 **주석 처리**된 곳 존재 → 폰트 미등록 소지.
- **영향**: shinylive(WASM)에서 한글 축/제목 라벨이 깨지거나(□□□) 폴백으로 의도와 다르게 표시될 위험. 앱마다 결과 상이 → UX 불균일.
- **확정 상태**: 정적 근거(설정 비일관)는 확정. 실제 깨짐 픽셀 검증은 §6 후속 과제.

### 3.3 빌드 산출물·저장소 위치 — 🟠 중간
- **현상**:
  - 빌드 산출물 `docs/`(77MB, 278개 파일)와 `_freeze/`(28개)가 git에 커밋됨.
  - 저장소가 **iCloud Drive**(`~/Library/Mobile Documents/com~apple~CloudDocs/`) 내부에 위치.
- **영향**:
  - `docs/`는 GitHub Pages 특성상 커밋이 불가피하나, 77MB가 소스와 뒤섞여 diff·clone 비대화. 소스 리뷰 시 노이즈.
  - iCloud 실시간 동기화 + `.git` 동시 접근은 **인덱스/객체 손상, 동기화 충돌** 위험. 개발 저장소를 클라우드 동기화 폴더에 두는 것은 일반적으로 권장되지 않음.

### 3.4 죽은 애널리틱스 설정 — 🟠 중간
- **현상**: `google-analytics: "UA-229551680-1"`.
- **영향**: 표준 **Universal Analytics(UA-)는 2023-07-01자로 신규 hit 처리 중단**, 2024-07-01자로 UA 인터페이스·API 접근도 종료(Google 공식 안내). 따라서 이 UA 속성만으로는 지표 수집이 되지 않아 **GA4(`G-`) 마이그레이션 필요**. (단 "전혀 수집 안 됨"의 최종 판정은 배포 HTML의 connected site tag·별도 태그 유무 확인 후 — §6.)

### 3.5 라이선스 표기 모순 — 🟠 중간
- **현상**: `license: "CC BY-SA-NC & GPL-3"`.
- **영향**: 코드(GPL-3)와 콘텐츠(CC BY-SA-NC)를 대상별로 분리 적용한다면 그 자체가 모순은 아니나, **현재 저장소는 그 분리를 문서화하지 않음**(루트에 `LICENSE`/`NOTICE` 없음). 참고로 FSF는 CC의 **NonCommercial(NC)** 조항을 자유 문서 라이선스로 인정하지 않으며, CC BY-SA 4.0은 GPLv3로의 **일방향** 호환만 인정. 결과적으로 표기가 불명확해 재사용자에게 혼란을 줌. 표준 SPDX 표기도 아님.

### 3.6 문서화 부재 — 🟠 중간
- **현상**: `README.md`가 사실상 비어 있음(`# BitStat2` 한 줄). `CONTRIBUTING`, 빌드/배포 절차, 로컬 개발 가이드 없음.
- **영향**: 신규 기여자가 렌더 방법(`quarto render`), 배포 흐름, 폴더 규칙을 알 수 없음. 오픈소스 협업 진입장벽.

### 3.7 CI/CD 부재 — 🟠 중간
- **현상**: `.github/` 없음. 렌더 후 `docs/`를 **수동 커밋**하는 방식으로 추정.
- **영향**: 렌더 누락·부분 렌더로 소스와 배포본 불일치 가능. 링크 깨짐·빌드 실패를 자동 검출하지 못함. 77MB 산출물을 사람이 커밋.

### 3.8 WASM 로딩 성능 리스크 — 🟡 낮음
- **현상**: 일부 앱이 `tidyverse` 전체를 `library(tidyverse)`로 로드(2개). 앱별로 `GGally`, `esquisse`, `plotly`, `vcd` 등 무거운 패키지 로드.
- **영향(정정)**: shinylive/webR은 R 패키지를 **브라우저에서 소스 컴파일하지 않음**. 사전 컴파일된 **WebAssembly 바이너리 패키지를 다운로드해 가상 파일시스템에 설치·로드**하는 방식(저장소의 `qwebr-document-engine-initialization.js`도 `webr::install()` 호출). 따라서 리스크는 "컴파일"이 아니라 **초기 자산·바이너리 다운로드 용량, 가상 FS 설치 시간, 메모리 사용량**. 무거운 메타패키지·다수 의존성일수록 초기 로딩 지연. 필요한 하위 패키지만 로드하면 개선.

### 3.9 webr 설정·메타 이슈 — 🟡 낮음 (⚠️정정)
- **정정**: 초안에서 "`{webr-r}` 셀 0개"라 기술했으나 **사실 오류**였음(검토용 grep이 zsh glob 오류로 실패한 것을 오독). **실제로는 24개 파일에 걸쳐 `{webr-r}` 블록 27개가 존재**하며 webr는 활발히 사용 중. 따라서 webr는 죽은 설정이 아님.
- 오히려 확인할 것: `_quarto.yml`의 `webr.packages` 목록이 실제 `{webr-r}` 셀에서 쓰는 패키지( `openintro`, `showtext`, `gt` 등)를 모두 커버하는지 — 누락 시 셀 실행 실패 가능(→ §6 후속).
- `pyodide` 필터는 주석 처리 상태로 잔존(정리 대상).
- `styles.css`에 `color: blue` 하드코딩된 리스팅 제목 → 테마(spacelab)·접근성 대비와 별개로 관리되는 매직값.

### 3.10 재현성·외부 의존성 — 🟠 중간 (초안 누락, 적대적 검토로 추가)
- **현상**:
  - 랜덤 시뮬레이션 다수(`runif`/`sample`/`rnorm`)인데 `set.seed`는 **전체 23개 중 1개 모듈**에서만 사용. 예: `07_theory/sample_size`는 `rnorm(1000000)` 생성.
  - `03_viz/viz`는 렌더 시 외부 URL에서 CSV를 **`download.file`로 실시간 다운로드**(`bit2r/bitData` gist). 원격 소스 변경/삭제 시 빌드 깨짐.
  - 루트에 `renv.lock`/`pak.lock`/`DESCRIPTION` 등 **패키지 버전 고정 없음**.
- **영향**: 교육용 앱의 "매번 달라지는 상호작용"은 의도일 수 있으나, **정적 문서 렌더의 재현성**은 시드·데이터 스냅샷·버전 고정 부재로 보장되지 않음. `freeze`는 이를 부분적으로만 가려줌.

### 3.11 사용자 업로드 입력 검증(보안/견고성) — 🟠 중간 (초안 누락)
- **현상**: **8개 모듈**이 `fileInput` + `read.csv(input$file$datapath)`로 사용자 CSV를 업로드받음(`esquisse`, `viz`, `corr_reg`, `reg`, `dist_fitting`, `multivariate_stat`, `univariate_stat`, `clustering`).
- **영향**: 파일 크기·인코딩·열 개수·타입 검증 없이 바로 읽어 플롯 → 비정상/대용량 CSV에서 앱 크래시·먹통 위험. WASM은 클라이언트 실행이라 서버 유출 위험은 낮으나, **개인정보 업로드 금지 안내 부재**·오류 메시지 미처리로 UX 취약.
- **비고**: 하드코딩 시크릿 패턴은 이번 표적 검색에서 미발견.

### 3.12 접근성 — 🟠 중간 (초안에서 후속 언급 → 본문 승격)
- **현상**:
  - 다수 페이지가 `image: thumbnail.png`만 있고 `image-alt`/`fig-alt` **대체텍스트 없음**(`x_score`, `esquisse` 등).
  - shinylive iframe이 `viewerWidth: 800` **고정폭** → 모바일 가로 스크롤·잘림.
  - `styles.css`가 링크/카드 제목을 단순 `blue`/`color: blue`로 하드코딩 → 색 대비(WCAG)·다크모드 미고려.
- **영향**: 스크린리더·모바일·저시력 사용자 접근성 저하. 공익·포용을 표방하는 프로젝트(§about) 취지와 상충.

## 4. 향후 수정·추진 방안 (Roadmap)

### 4.1 즉시(1주 내) — 정확성/신뢰 회복
1. **폰트 표준화(§3.2)**: 전 앱 공통 폰트 부트스트랩 헬퍼 도입. shinylive/WASM에서 동작하는 방식으로 한글 폰트를 명시적 등록(`sysfonts::font_add` + 웹폰트, 또는 검증된 `showtext` 패턴)하고 `Tahoma` 제거. **브라우저에서 한글 라벨 실제 렌더 검증 필수.**
2. **애널리틱스 교체(§3.4)**: `UA-` → GA4 `G-XXXX`로 교체하거나, 미사용이면 설정 제거.

### 4.2 단기(1개월) — 유지보수성
4. **중복 제거(§3.1)**: `app.R`를 정본(single source of truth)으로 삼고, `index.qmd`는 `{shinylive-r}` 대신 **`include`/파일 참조 방식**으로 앱을 임베드하거나, 반대로 `index.qmd` 인라인을 정본으로 두고 `shiny/` 폴더 제거. 한 곳만 유지하도록 리팩터링 + 어느 쪽이 정본인지 README에 명시.
5. **README/CONTRIBUTING 작성(§3.6)**: 로컬 렌더(`quarto preview`), 배포 흐름, 모듈 추가 규칙, 폰트/패키지 컨벤션 문서화.
6. **CI 도입(§3.7)**: GitHub Actions로 `quarto render` 자동화 → `gh-pages` 브랜치 배포. 링크 체크(`quarto check`, lychee) 추가. 성공 시 §3.3의 소스-산출물 혼재 완화(산출물을 소스 브랜치에서 제거 가능).

### 4.3 중기(분기) — 구조/성능/재현성
7. **저장소 이전(§3.3)**: iCloud 동기화 폴더 밖(예: `~/dev/BitStat2`)으로 이전하여 `.git` 손상·충돌 위험 제거.

## 7. 검토의 적대적 재검증 (Adversarial Self-Review)

본 검토가 "제대로 된 기술검토"인지 스스로 반박·검증한 결과:
- **반박 시도 1 — "중복은 shinylive 표준 패턴 아닌가?"**: shinylive는 인라인 블록만으로 동작 가능. `shiny/app.R`는 필수가 아니며, 실제로 stale 사본으로 방치됨(§3.1 실측). → 지적 유효.
- **반박 시도 2 — "폰트는 앱마다 필요 없을 수도"**: 한글 라벨을 그리는 통계 플롯에서 폰트 미설정 8개는 폴백 위험 실재. Tahoma는 명백한 오류. → 유효(단, 픽셀 확정은 §6).
- **반박 시도 3 — "docs 커밋은 GitHub Pages 정상 관행"**: 맞음. 그래서 심각도를 🟠로 하향하고, 문제의 핵심을 iCloud 위치(§3.3)로 재초점. → 과장 교정 완료.
- **미확정 항목의 정직한 표기**: §3.4(UA 종료)는 일반 사실이나 이 사이트 실제 수집 중단은 대시보드 미확인 → "후속 확인"으로 분류. §3.8 성능은 미측정 → 🟡 유지.
- **결론**: 정적 근거로 확정 가능한 항목(3.1, 3.2 설정, 3.3, 3.4, 3.5, 3.6, 3.7, 3.10, 3.11, 3.12)과 실행 검증이 필요한 항목(3.2 픽셀, 3.8, 3.9)을 구분해 표기함으로써 과잉 단정 없이 검토의 타당성을 확보.
